Module 1

Le Lab Blocks Master

Le Lab est l'environnement de développement intégré (IDE) de SiteMaster. Il vous permet de concevoir, tester et gérer vos blocs Gutenberg directement depuis votre navigateur, sans ouvrir un terminal ou votre IDE local.

Créer son premier bloc

Le processus est volontairement minimal :

  1. Donnez un Nom à votre bloc (ex: "Hero Slider").
  2. Optionnel : choisissez une Icône (Lucide) et une Catégorie Gutenberg.
  3. Écrivez votre Code PHP/HTML dans l'éditeur.
  4. Cliquez sur Enregistrer.

Le bloc est instantanément disponible dans Gutenberg, sans rechargement de page. Aucune compilation, aucun build.

Exemple de bloc minimaliste

<div class="tw-bg-slate-900 tw-p-10 tw-rounded-none">
 <h1 class="tw-text-white tw-text-4xl tw-font-bold">
 Bonjour depuis mon bloc
 </h1>
 <p class="tw-text-slate-400 tw-mt-3">
 Rendu via Blocks Master, en PHP pur.
 </p>
</div>

Exemple avec PHP dynamique

<div class="tw-p-6">
 <h1><?= get_the_title() ?></h1>
 <p>Publié le <?= get_the_date() ?></p>
 <?= get_the_content() ?>
</div>

Stockage & Synchronisation (Hybride)

Blocks Master v3.0 utilise un système de stockage hybride :

  • Base de Données (wp_blocks_master) : votre code est stocké en DB pour garantir l'intégrité et permettre le versionnement.
  • Filesystem (/blocks/*.php) : une copie physique est générée à chaque sauvegarde.
  • Rendu : WordPress inclut directement le fichier physique, ce qui est aussi rapide que du code natif inclus dans un thème.
⚠️ ATTENTION

Ne modifiez pas les fichiers dans /blocks/ directement via FTP ou un éditeur de fichiers. Vos changements seraient écrasés par la base de données lors de la prochaine sauvegarde dans le Lab.

Historique des Versions

Le Lab conserve automatiquement les 5 dernières versions de chaque bloc. En bas de l'éditeur, une liste vous permet de consulter et de restaurer un état précédent en un clic.

Si vous avez cassé votre bloc ou souhaitez revenir à un design précédent, ouvrez l'onglet "Historique" dans le Lab et cliquez sur "Restaurer" à côté de la version cible.

Attributs de Bloc (Paramètres Gutenberg)

Le système d'attributs transforme un bloc statique en un composant éditable directement depuis la barre latérale de Gutenberg — sans quitter le Lab.

Dans l'onglet "Attributs", pour chaque attribut, vous définissez :

  • Clé : L'identifiant utilisé dans le code (ex: couleur_fond, titre).
  • Label : Le nom affiché dans la barre latérale de Gutenberg ou d'Elementor.
  • Type :
    • text — Texte court (TextControl Gutenberg)
    • number — Nombre ou slider (RangeControl)
    • boolean — Case à cocher / Toggle
    • color — Palette couleur
    • select — Liste déroulante
    • image — Sélecteur médias WordPress
  • Défaut : La valeur utilisée si aucune valeur n'a été saisie par l'éditeur.

Récupérer un attribut dans le code

Tous les attributs définis sont injectés dans la variable $attributes, disponible globalement dans votre code de bloc :

<?php
$titre = esc_html($attributes['titre'] 'Titre par défaut');
$couleur = esc_attr($attributes['couleur_fond'] '#4f46e5');
$lien_image = esc_url($attributes['image'] '');
?>

<div style="background-color: <?= $couleur ?>">
 <h1 class="tw-text-white"><?= $titre ?></h1>
 <?php if ($lien_image): ?>
 <img src="<?= $lien_image ?>" class="tw-w-full tw-rounded-none" alt="">
 <?php endif; ?>
</div>

Support Tailwind CSS

Le plugin embarque Tailwind CSS compilé en local. Pour éviter tout conflit avec le thème ou le back-office WordPress, vous devez utiliser le préfixe tw- sur toutes vos classes.

❌ Incorrect

class="bg-blue-500 rounded p-4"

✅ Correct

class="tw-bg-blue-500 tw-rounded tw-p-4"
💡 TIP

Si vous utilisez des valeurs arbitraires Tailwind (ex: tw-w-[423px]) et que le style ne s'applique pas, privilégiez un attribut style="..." inline. Plus fiable dans l'iframe de l'éditeur Gutenberg.

Sécurité Serveur — DISALLOW_FILE_EDIT

Pour les environnements de production hautement sécurisés, Blocks Master respecte les directives WordPress.

⚠️ WARNING

Si la constante DISALLOW_FILE_EDIT est activée dans votre wp-config.php, l'accès au Lab est automatiquement bloqué. C'est une décision intentionnelle — si votre serveur interdit les modifications de fichiers via le dashboard, le Lab le respecte.

// Dans wp-config.php
define('DISALLOW_FILE_EDIT', true);

Beaucoup d'hébergeurs activent cette constante par défaut pour des raisons de sécurité. Si votre Lab est bloqué sans raison apparente, vérifiez ce fichier en premier.

Support Elementor

Si Elementor est installé sur le site, chaque bloc créé dans le Lab est automatiquement disponible dans le panel de widgets, dans la catégorie "SiteMaster". Aucune configuration requise.

  • Les attributs que vous avez définis deviennent des contrôles Elementor natifs (Color Picker, Toggle, Number, etc.).
  • Le rendu est identique à Gutenberg — le même fichier PHP est inclus.
  • Le widget porte le nom de votre bloc et son icône eicon-code.

Régénération manuelle des fichiers

Si vos fichiers /blocks/*.php sont perdus (migration, restauration de DB), un bouton "🔄 Régénérer tous les fichiers" est disponible dans la barre latérale du Lab. Il recrée tous les fichiers depuis la base de données en un clic.

💡 TIP

La régénération est utile après une migration de base de données. La DB est la source de vérité — les fichiers ne sont que son miroir de performance.
